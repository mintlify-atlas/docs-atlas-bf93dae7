Manage user accounts, assign roles, and control access permissions for the Black Michi Studio platform.

## Overview

The user management system allows admins to view all registered users, modify user roles, and manage user accounts. All user operations require admin authentication.

<Warning>
  All user management operations require:
  - Valid JWT authentication token
  - User role must be `admin`
  - Both `requireAuth` and `requireAdmin` middleware
</Warning>

## User Schema

Users are stored in the `usuarios` table with the following structure:

| Field | Type | Description |
|-------|------|-------------|
| `id` | SERIAL | Primary key |
| `nombre` | VARCHAR(100) | User's full name |
| `email` | VARCHAR(100) | Email address (unique) |
| `password` | VARCHAR(255) | Bcrypt hashed password |
| `rol` | VARCHAR(20) | User role: `admin` or `cliente` |
| `telefono` | VARCHAR(20) | Phone number |
| `direccion` | TEXT | Address |
| `email_verified` | BOOLEAN | Email verification status |
| `created_ip` | VARCHAR(45) | Registration IP address |
| `activo` | BOOLEAN | Account active status |
| `created_at` | TIMESTAMP | Registration date |
| `updated_at` | TIMESTAMP | Last update timestamp |

Defined in `backend/schema.sql:15-28`.

## User Roles

The platform supports two user roles with different permission levels:

### Role Types

| Role | Value | Description | Permissions |
|------|-------|-------------|-------------|
| Admin | `admin` | Administrator | Full access to all admin endpoints and operations |
| Client | `cliente` | Customer | Access to customer-facing features only |

<Note>
  Role values are enforced via database CHECK constraint: `CHECK (rol IN ('admin', 'cliente'))`
</Note>

### Default Role

New users are assigned the `cliente` role by default during registration:

```sql
rol VARCHAR(20) DEFAULT 'cliente'
```

## Authentication and Authorization

The admin authentication system uses JWT tokens with role-based access control.

### Authentication Middleware

Implemented in `backend/middleware/auth.js`:

#### requireAuth Middleware

Validates JWT token and loads user data:

```javascript
async function requireAuth(req, res, next) {
  const auth = req.headers.authorization;
  if (!auth) return res.status(401).json({ error: 'No autorizado' });

  const token = auth.split(' ')[1];
  try {
    const decoded = jwt.verify(token, JWT_SECRET);

    // Load full user data from database
    const result = await pool.query(
      'SELECT id, nombre, email, rol FROM usuarios WHERE id = $1',
      [decoded.id]
    );

    if (!result.rows.length) {
      return res.status(401).json({ error: 'Usuario no encontrado' });
    }

    req.user = result.rows[0]; // Sets id, nombre, email, rol
    next();
  } catch (err) {
    res.status(401).json({ error: 'Token inválido' });
  }
}
```

#### requireAdmin Middleware

Verifies user has admin role:

```javascript
function requireAdmin(req, res, next) {
  if (!req.user) {
    return res.status(401).json({ error: 'No autenticado' });
  }
  if (req.user.rol !== 'admin') {
    return res.status(403).json({ error: 'Solo admin' });
  }
  next();
}
```

### Middleware Chain

All admin routes use both middleware functions:

```javascript
router.get(
  "/usuarios",
  requireAuth,      // 1. Verify token and load user
  requireAdmin,     // 2. Verify admin role
  adminController.getAllUsers
);
```

## Managing Users

### Viewing All Users

<Steps>

Implemented in `backend/controllers/adminController.js:93-107`:

```javascript
exports.getAllUsers = async (req, res) => {
    const result = await pool.query(`
```

### Updating User Roles

<Steps>
  <Step title="Select user">
    Identify the user whose role you want to change.

    ```javascript
    const userId = 5;
    ```
  </Step>

  <Step title="Choose new role">
    Select either `admin` or `cliente`.

    ```javascript
    const newRole = "admin"; // or "cliente"
    ```
  </Step>

  <Step title="Submit role update">
    Call the update endpoint with the new role.

    ```javascript
    await api.put(`/admin/usuarios/${userId}/rol`, {
      rol: newRole
    });
    ```
  </Step>

  <Step title="Verify update">
    The user's permissions will immediately reflect the new role.
  </Step>
</Steps>

Implemented in `backend/controllers/adminController.js:109-125`:

```javascript
exports.updateUserRole = async (req, res) => {
    const { id } = req.params;
    const { rol } = req.body;

    await pool.query(
        `UPDATE usuarios SET rol = $1 WHERE id = $2`,
        [rol, id]
    );

    res.json({ ok: true });
};
```

<Warning>
  Changing a user's role immediately affects their access permissions. Admin users gain full access to all admin endpoints, while cliente users lose admin access.
</Warning>

### Deleting Users

<Warning>
  **DESTRUCTIVE OPERATION**: Deleting a user permanently removes their account and may affect:
  - Order history (orders will have null `usuario_id`)
  - Any user-created content
  - Authentication tokens will be invalidated
  
  This action cannot be undone.
</Warning>

<Steps>
  <Step title="Confirm deletion">
    Always show a confirmation dialog with clear warnings.

    ```javascript
    const confirmDelete = window.confirm(
      `Are you sure you want to delete user ${userName}? This cannot be undone.`
    );
    if (!confirmDelete) return;
    ```
  </Step>

  <Step title="Call delete endpoint">
    Send DELETE request with user ID.

    ```javascript
    await api.delete(`/admin/usuarios/${userId}`);
    ```
  </Step>

  <Step title="Refresh user list">
    Reload the user list to reflect the deletion.

    ```javascript
    await fetchUsers();
    ```
  </Step>
</Steps>

Implemented in `backend/controllers/adminController.js:127-139`:

```javascript
exports.deleteUser = async (req, res) => {
    const { id } = req.params;

    await pool.query(`DELETE FROM usuarios WHERE id = $1`, [id]);

    res.json({ ok: true });
};
```

## Database Constraints

The usuarios table enforces several important constraints:

### Unique Email

```sql
email VARCHAR(100) NOT NULL UNIQUE
```

Each email address can only be registered once.

### Role Validation

```sql
rol VARCHAR(20) DEFAULT 'cliente' CHECK (rol IN ('admin', 'cliente'))
```

Only `admin` and `cliente` values are accepted.

### Foreign Key Relationships

Users are referenced by:
- `pedidos.usuario_id` - Orders placed by user (SET NULL on delete)
- `valoraciones.usuario_id` - Product reviews (SET NULL on delete)

<Note>
  When a user is deleted, their orders and reviews remain in the database with null `usuario_id` to preserve historical data.
</Note>

## Security Best Practices

1. **Never expose passwords** - API responses exclude password hashes
2. **Validate role changes** - Ensure only valid roles are assigned
3. **Confirm destructive actions** - Require confirmation for user deletion
4. **Audit admin actions** - Log role changes and deletions
5. **Protect admin accounts** - Prevent admins from deleting themselves
6. **Use HTTPS** - Always transmit credentials over secure connections

## Permission Levels

### Cliente (Customer) Permissions

- View and edit own profile
- Place orders
- View own order history
- Submit product reviews
- Update shipping address

### Admin Permissions

- All cliente permissions
- View all users
- Modify user roles
- Delete user accounts
- Manage products (create, update, delete)
- Manage orders (view all, update status)
- View dashboard statistics
- Access admin panel

<Note>
  Admin permissions are enforced at the route level using the `requireAdmin` middleware. No additional role checks are needed in controller logic.
</Note>

## Automatic Timestamps

The `updated_at` field is automatically updated via database trigger:

```sql
CREATE TRIGGER update_usuarios_updated_at BEFORE UPDATE ON usuarios
FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

This ensures accurate tracking of user account modifications.
